## Method (and an honesty note about the reviewer model)

The app was **launched and driven as a real user against a real, populated
backend** — never from code alone. Two complementary live techniques were used
(neither mocks product behaviour; the only mock is the sanctioned Secret
Service stand-in):

1. **Live GTK release binary** under Xvfb `:55` (1440×900×24) +
   `scripts/run-with-mock-keyring.sh`, screenshotted with ImageMagick
   `import -window root`. Proves the app truly runs on this box, reaches the
   first-run wizard, and talks to the seeded backend with **zero errors**.
2. **Flutter integration probes** (in `/tmp/opencode/probe_round5/`, **outside**
   the repo so `lib/`/`test/` are untouched) that pump the **production**
   `ScriptDetailsDialog` / `UnifiedSetupWizard` / `AccountProfileScreen` against
   the **real** `MarketplaceOpenApiService` hitting the **real** local backend
   (no `MockClient`, no fake service — real Ed25519 keypairs via the FFI bridge).
   They print authoritative **visible-text dumps** (the primary structural
   evidence) and capture render-tree screenshots.

> **Reviewer constraint (honesty):** this model **cannot view images** (the
> `Read` tool rejects image input). The screenshots in
> [`ux_screenshots/round5/`](ux_screenshots/round5/) are captured as evidence
> for the human reader; the reviewer's own analysis is grounded in the probes'
> visible-text dumps + the source (`file:line`) + the live app log. This is the
> same discipline Round 4 documented. Where a screenshot's exact pixel state
> could not be textually confirmed, it is stated.

# Surface reviews

## 1. First-run wizard (NEW USER) — `unified_setup_wizard.dart`

**Evidence:** probe `D_wizard.txt` (text dump) + `D_wizard.png` +
live GTK `ux_shot_01_main.png`.

The new-user wizard is a **single, clean screen** — minimal friction:

```
Create Your Profile
Set up your profile to start creating and running scripts.
Display Name          [ How should we call you? ]
Username (optional)
Marketplace           Create a marketplace account to share scripts
                      and interact with the community
                      [ alice_dev ]
Username requirements: 3-32 chars · Lowercase · a-z, 0-9, _, -
                       [ Get Started ]
```

**What's good:** display name + **optional** marketplace username on one
screen; username requirements shown **inline, up-front** (charset, length,
case) — this pre-empts the most common submission errors; StorageReady under
the mock keyring is reached transparently (WU-S2 works).

**Friction / polish:**
- The label **"Marketplace"** floating above the optional username field is
  terse and slightly disconnected from "Username (optional)". A reader has to
  infer that the field *is* the marketplace username. (Minor — → UXR5-5.)
- No visible progress/step affordance, but none is needed (it's one step).
  Honest single-screen — better than a fake multi-step wizard.

## 2. Marketplace browse + Script details (the UX-6 work)

**Evidence:** probe `A1_free_details.txt` / `A2_paid_details.txt` +
`A1_free_details.png` / `A2_paid_details.png`; backend `trending` API dump.

### Details dialog — works. The Round-4 UX-5/UX-6 finish is solid live.

**FREE** (`A1` dump): title, "by {author}", "Development" category badge,
**"FREE"** price badge, "No rating", tabs **Details / Reviews / Versions**,
Description, Tags, **Code Preview** pane that **renders the real source**
(UX-6: from `/preview`, not a full download), Statistics (Downloads / Rating /
Version 1.0.0 / Updated "3 days ago"), and tooltips **"Close (Esc)"** +
**"Switch tabs with ← / → arrows"** + **"Copy preview"**.

**PAID** (`A2` dump): same shell with a green **"$1.00"** price badge, the
preview pane shows the **server-capped teaser** (`-- Updated source`) — the
paid bundle is **never** fetched for preview (confirmed) — and a primary
**"Buy for $1.00"** CTA (UX-5b). No dead-end for paid scripts.

### ICPay degraded state (503) — HONEST and CLEAR ✅

ICPay is unconfigured (`GET /payments/icpay/config` → **HTTP 503**
`{"error":"ICPAY_PUBLISHABLE_KEY not configured"}`). The degraded path is
handled correctly:
- `marketplace_open_api_service.dart:460-461` throws
  `PaymentsNotConfiguredException` on the 503;
- `_buyScript` (`scripts_screen.dart:625-633`) catches it and shows a **red
  SnackBar: "Payments are not configured on this server yet. Try again later."`;
- the backend **warns at startup** (`warn_if_icpay_unconfigured`, per
  `HUMAN_EXPECTATIONS.md` §4).

This is exactly the loud-degradation the project values. The only nit is copy:
**"Try again later"** implies a transient blip, but an unconfigured provider is
a server-side gap the user cannot fix by retrying (→ UXR5-4, P2).

### Headline finding — the marketplace is seeded with test fixtures (slop)

The 20 trending scripts are obvious **test seed data**, not premium content:

```
'My Test Script for Publishing' (price 0)
'Updated Title' (price 1.0)          ← the PAID script is literally named "Updated Title"
'List Test Script 1', 'List Test Script 2'
'Unique Searchable Title', 'Development Script', 'Utility Script', 'Public Script'
```

For a product positioned as a **"premium marketplace"**, shipping a browse
surface where the only paid script is called *"Updated Title"* with description
*"Updated description"* is **slop that undercuts the premium positioning** the
moment a new user opens the Scripts tab. (→ UXR5-1, **P1** — content, not code.)

### Headline finding — language/metadata mismatch (TypeScript badge over Lua)

The FREE script's preview pane shows a **"TypeScript"** language badge, but the
rendered source is **Lua**:

```
Code Preview
TypeScript
function init(arg)        ← Lua syntax
  return { message = "...", count = 0 }, {}
end
function view(state) ... function update(msg, state) ...
```

The `/preview` endpoint returns `language: "typescript"` while the actual
bundle is the **old Lua autorun runtime**. A user reading "TypeScript" above
Lua code loses trust in the metadata. (→ UXR5-2, **P1** — either validate
language at upload or detect from content.)

### Secondary — 45 s marketplace timeout is too long for browse

`MarketplaceOpenApiService._timeout = Duration(seconds: 45)`
(`marketplace_open_api_service.dart:69`) applies to **every** call, including
browse/search/trending. On a slow or briefly-unreachable backend a new user
stares at a spinner for up to **45 seconds** before any error — the brief
explicitly flags spinners >a few seconds. (→ UXR5-3, **P1**.)

## 5. Error states (TD-5 categorized display)

**Evidence:** source `utils/error_categories.dart:1-154` (the TD-5 landing) +

# Concrete UX-2 proposals (this round's derived work units)

Each is a candidate WU for `UI_EXCELLENCE_PLAN.md`, in the established format.
Priorities: **P1** = high-friction/honesty, **P2** = polish. Effort: **S/M**.

### UXR5-1 — Ship a curated (or honestly "demo") marketplace seed, not test fixtures  [P1, M]
- **Problem:** The first thing a new user sees on the Scripts tab is 20 obvious
  test fixtures (`List Test Script 1/2`, `Updated Title`, `Public Script`).
  The sole paid script is named *"Updated Title"* with description *"Updated
  description"*. This is exactly the "no slop / no placeholders shipped as a
  feature" bar (`HUMAN_EXPECTATIONS.md` §2) violated at the content layer.
- **Grounding:** `GET /api/v1/scripts/trending?limit=20` (measured 2026-07-08)
  returns the titles above; probe `A1`/`A2` render them verbatim.
- **Concrete change:** Replace the seed with a small set of **real, useful,
  well-described scripts** (a "Hello IC" starter, an NNS-balance reader, a
  ledger-transfer helper — the kinds already referenced in the Canisters tab).
  If real content isn't ready, mark each demo row with an explicit **"Demo"**
  badge so it reads as intentional, not abandoned. Coordinate with `backend/`
  seed scripts.
- **Dependencies / Risk:** backend seed data / **LOW** (content, not code).
- **Acceptance:** a new user's first browse shows ≥5 scripts with real titles +
  descriptions (or clearly-badged demos); no "Updated Title"-style fixtures.

### UXR5-2 — Make script `language` honest (validate/detect, don't trust stale metadata)  [P1, M]
- **Problem:** The details dialog shows a **"TypeScript"** badge above **Lua**
  source. The `/preview` `language` field disagrees with the actual bundle, so
  the language label actively misleads.
- **Grounding:** probe `A1` dump — `Code Preview / TypeScript` over
  `function init(arg) … end` (Lua). Backend `/preview` returns
  `language:"typescript"` for a Lua bundle.
- **Concrete change:** At upload (`quick_upload_dialog` / backend), **validate
  or detect** the language from the bundle (a small heuristic: `function … end`
  + `local` → Lua; `export`/`: type` → TypeScript) and reject/flag a mismatch.
  If detection is unreliable, prefer showing **no language badge** over a wrong
  one. Single-source the label from the detected value.
- **Dependencies / Risk:** touches upload path + backend metadata / **MED**.
- **Acceptance:** the badge matches the bundle's actual syntax for seeded + new
  scripts; a mismatch at upload is flagged. New test covers Lua-vs-TS detection.

### UXR5-3 — Shorter, per-call marketplace timeouts (45 s → ~8–10 s for browse)  [P1, S]
- **Problem:** `MarketplaceOpenApiService._timeout = Duration(seconds: 45)`
  (`marketplace_open_api_service.dart:69`) applies to trending/search/details,
  so a slow or briefly-unreachable backend holds the UI in a spinner for up to
  **45 s** before any error surfaces.
- **Grounding:** `marketplace_open_api_service.dart:69` (single `_timeout`,
  every call).
- **Concrete change:** Introduce two tokens via the existing `AppDurations`
  pattern — a **browse timeout (~8 s)** for list/search/preview and a **download
  timeout (keep ~45 s)** for full-bundle fetches. Surface the categorized error
  (TD-5 `network`/`server`) promptly on timeout.
- **Dependencies / Risk:** none / **LOW**.
- **Acceptance:** a dead backend errors the browse view in ≤10 s (not 45); full
  downloads keep the long budget; `just test-feature marketplace` green.

### UXR5-4 — Sharpen the ICPay-degraded copy  [P2, S]
- **Problem:** The 503 SnackBar says *"Payments are not configured on this
  server yet. **Try again later.**"* — "try again later" implies a transient
  blip, but an unconfigured provider is a server-side gap the user cannot fix.
- **Grounding:** `scripts_screen.dart:625-633`.
- **Concrete change:** Reword to something honest and non-transient, e.g.
  *"Paid purchases aren't available in this environment yet."* (optionally +
  *"You can still download free scripts."*). Drop the "try again later" nudge.
- **Dependencies / Risk:** none / **TRIVIAL** (copy).
- **Acceptance:** SnackBar copy updated; existing buy-path test assertion updated.

### UXR5-5 — Wizard: connect the "Marketplace" label to the username field  [P2, S]
- **Problem:** The label **"Marketplace"** sits between "Username (optional)"
  and the field; a new user must infer the field *is* the marketplace username.
- **Grounding:** probe `D` dump — `Username (optional)` / `Marketplace` /
  description / `[ alice_dev ]`.
- **Concrete change:** Inline the qualifier, e.g. field label
  **"Marketplace username (optional)"**, and drop the free-floating
  "Marketplace" line (fold it into the description if needed).
- **Dependencies / Risk:** none / **TRIVIAL**.
- **Acceptance:** the field's purpose is unambiguous from its label alone.

### UXR5-6 — Tell a local-only user where passkeys appear  [P2, S]
- **Problem:** A local-only (unregistered) user sees no passkey surface and no
  hint of when/where passkey setup becomes available.
- **Grounding:** probe `C` dump — no passkey tile for `account: null`.
- **Concrete change:** In the local-only account screen, show a disabled/hint
  passkey row: *"Passkeys — available after you register an account (browser
  only on Linux)."* Reuses the honest `PasskeyPlatform` degradation copy.
- **Dependencies / Risk:** none / **LOW**.
- **Acceptance:** local-only users see an explanatory passkey hint, not silence.

### UXR5-7 — Let the app read its API endpoint at runtime (dev), not only at build  [P2, S — devex/honesty]
- **Problem:** `AppConfig._apiEndpoint` is a build-time `String.fromEnvironment`
  defaulting to `https://icp-mp.kalaj.org` (`app_config.dart:5-8`). The env var
  `MARKETPLACE_API_PORT` — implied by tooling/`just` and this task's brief to
  steer the app at runtime — is **ignored** by the shipping binary. The first
  live launch therefore hit the remote host and got Cloudflare 530.
- **Grounding:** `app_config.dart:5-21` (no `Platform.environment` read); the
  live app log before rebuild: `Failed to load marketplace scripts: HTTP 530`.
- **Concrete change:** In dev/debug builds, let an env override
  (`PUBLIC_API_ENDPOINT` / `MARKETPLACE_API_PORT`) take precedence over the
  compiled default (guarded by `kDebugMode` so release is unchanged). Keep the
  build-time default for release.
- **Dependencies / Risk:** none / **LOW** (dev-only branch; release unaffected).
- **Acceptance:** in debug, `export MARKETPLACE_API_PORT=N` redirects the app
  without a rebuild; release behavior unchanged.
